# Coding Protocol for Policy Intent Drift Corpus

Status: locked coding protocol, release release, 2026-06-20. The admitted corpus, source spans, roles, and validation gates are fixed for BEP-Deep/BEP-Max reproduction.

## Unit of coding

The primary unit is a source-backed policy claim. A source-backed policy claim is a contiguous public-documentation, specification, release-note, issue, pull-request, or example statement that satisfies all of the following:

1. it refers to a browser-enforced security policy or generated response header;
2. it states an explicit intended effect, default, migration behavior, compatibility condition, or generated-header surface;
3. it can be linked to one locked source-manifest row;
4. it can be represented without private data or live external scanning.

The secondary unit is a deterministic fixture derived from one or more coded claims. A fixture must keep source identifiers and a derivation label.

## Inclusion criteria

Include a claim when it is public, source-stable enough to cite, and within one of the locked policy families: CSP, CORS, HSTS, Permissions-Policy, CORP, COOP, COEP, or framework-generated response headers for those families.

Include framework claims only when they describe documented behavior, generated defaults, documented configuration APIs, or officially documented migration/compatibility constraints.

Include issue or pull-request evidence only when it is public, not from a private/security embargo context, and directly concerns policy maintenance or generated policy behavior.

## Exclusion criteria

Exclude live website responses, private traffic, private repositories, commercial scanner output, real user accounts, hosted scanner APIs, exploit traffic, and examples that require unavailable credentials or nonpublic infrastructure.

Exclude source text whose policy claim cannot be separated from unrelated application-specific behavior.

Exclude claims where intent has to be inferred from tone, marketing language, or unstated developer expectations. Only explicit source text or directly generated headers may ground intent labels.

## Core codes

* `explicit_intent_claim`: source states the intended protection, migration mode, default behavior, compatibility property, or generated-header promise.
* `generated_policy_surface`: source or fixture states response headers generated by a framework, middleware, or layer.
* `browser_effective_semantics`: source or fixture records the effective outcome under the locked BEP-IR context.
* `semantic_conflict_witness`: a minimized header/context pair showing mismatch between explicit intent and browser-effective outcome.
* `negative_control`: a fixture or claim that should not produce a drift witness under the locked conflict relation.
* `unsupported_syntax`: syntax or behavior outside the locked semantic fragment, recorded separately and not recoded as drift.
* `wrapper_failure`: an external baseline could not run or did not emit parseable output; recorded separately and not recoded as agreement/disagreement.

## Intent-label assignment

Intent labels are assigned only from this locked set unless a protocol amendment adds a label before full analysis:

* `enforce_script_restriction`
* `nonce_based_strict_csp`
* `allow_credentialed_cors`
* `deny_public_credentialed_cors`
* `enforce_https_only`
* `cross_origin_isolation_without_embed_breakage`
* `header_generation_default`
* `migration_monitoring_before_enforcement`
* `negative_control_no_drift`

## Coding workflow

1. Record source row in `source_manifest.csv`.
2. Extract admitted-source claims into a claim table with source id, section or anchor, short paraphrase, policy family, and intent label.
3. Apply inclusion/exclusion rules.
4. Assign intent label and evidence type.
5. Derive fixture only using locked transformation rules.
6. Run BEP-IR oracle and baselines.
7. Record unsupported syntax, wrapper failure, and manual-audit notes without dropping the sample.

## Validation

The full experiment must include a two-pass validation process:

* Pass 1: coder assigns source rows, claim units, and intent labels from the locked schema.
* Pass 2: an independent audit pass re-checks at least 25% of claims, all positive witness classes, and all disagreement cases against the source snapshots.

Agreement is reported for intent-label assignment and inclusion/exclusion decisions. Disagreements are resolved by source text and protocol rules, not by experimental outcome.

## Failure handling

A fixture may be marked `unsupported_syntax`, `out_of_scope_context`, `baseline_unavailable`, or `baseline_parse_failure`. These statuses remain in denominators where applicable. No failed sample may be removed because it weakens a result.

## Provenance

Every coded claim and fixture must include source id, access date, version or release marker, and derivation rule. Deterministic generated fixtures are recorded as project-generated from public-source rules, not as external observations.
